**Part A: RLS policies**

After running `prisma migrate dev`, edit the generated migration SQL file to append RLS policies for ALL new tables that have a tenantId column. Follow the exact pattern from existing migrations (see 20260214000003_add_route_model/migration.sql and 20260215000001_add_fleet_intelligence_models/migration.sql).

For each of these tables: RouteExpense, ExpenseCategory, ExpenseTemplate, RoutePayment — append:
```sql
-- ==============================================
-- Row-Level Security Setup for {TableName}
-- ==============================================

ALTER TABLE "{TableName}" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "{TableName}" FORCE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation_policy ON "{TableName}"
  FOR ALL
  USING ("tenantId" = current_tenant_id())
  WITH CHECK ("tenantId" = current_tenant_id());

CREATE POLICY bypass_rls_policy ON "{TableName}"
  FOR ALL
  USING (current_setting('app.bypass_rls', TRUE)::text = 'on');
```

NOTE: ExpenseTemplateItem does NOT have a tenantId column directly (it inherits tenant isolation through its template's tenantId). However, since it's accessed through joins with ExpenseTemplate, no separate RLS needed. The RLS on ExpenseTemplate already protects template items because queries always go through the template.

After editing, apply the migration: `npx prisma migrate dev` (Prisma will detect the edited migration and apply it). If Prisma complains about drift, use `npx prisma migrate resolve --applied 20260216000001_add_route_finance_models` and then run `npx prisma db push` to sync.

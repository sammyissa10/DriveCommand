  <name>Task 2: Bundle size verification and optimization</name>
  <files>

  <action>
1. Run the production build with bundle analysis:
```bash
npx next build
```

After the build completes, Next.js outputs a route-by-route size analysis. Check the "First Load JS" column for each route. The success criteria requires first-load JS under 500KB.

2. If first-load JS exceeds 500KB, apply these optimizations:

**Priority 1: Dynamic import chart components on dashboard pages**
Recharts is currently statically imported in chart components. While chart components are 'use client', they're still bundled into the page's JS. To reduce first-load size:

In safety/page.tsx and fuel/page.tsx, wrap chart component imports with `next/dynamic`:
```tsx
import dynamic from 'next/dynamic';
const SafetyScoreCard = dynamic(() => import('@/components/safety/safety-score-card').then(m => ({ default: m.SafetyScoreCard })));
// etc. for each chart component
```

Note: This only makes sense if the build output shows these routes exceeding the budget. Server component pages that import client components will tree-shake the server-side code, but the client components themselves add to the bundle. Dynamic import defers them from first-load.

**Priority 2: Check for large dependencies in the bundle**
If `@turf/turf` is imported as a full package anywhere (it's ~500KB), ensure it's only imported in seed scripts (not app code). The live-map uses it but should already be behind the dynamic import boundary. Verify with:
```bash
# Check if turf is imported in app code (not seeds)
grep -r "from '@turf" src/ --include="*.ts" --include="*.tsx" | grep -v "node_modules"
```

**Priority 3: Verify Leaflet is behind dynamic import boundary**
Leaflet + react-leaflet should only load when the live-map page is visited. The live-map-wrapper.tsx already uses `dynamic(() => import('./live-map'), { ssr: false })`. Verify no other file imports leaflet directly.

3. After any optimizations, re-run `npx next build` and verify all routes show First Load JS < 500KB.

4. Record the final bundle sizes in the summary for documentation. Capture the build output showing route sizes.

If bundle is already under 500KB without changes (likely, since charts are client components behind page boundaries), document this finding and note that no optimization was needed.

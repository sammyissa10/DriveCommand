**File 1: `src/components/routes/use-unsaved-changes-warning.ts`**

Create a custom React hook that warns users before losing unsaved form changes:

```typescript
import { useEffect } from 'react';

export function useUnsavedChangesWarning(isDirty: boolean) {
  useEffect(() => {
    if (!isDirty) return;

    const handleBeforeUnload = (e: BeforeUnloadEvent) => {
      e.preventDefault();
      e.returnValue = ''; // Required for Chrome
      return ''; // Required for some browsers
    };

    window.addEventListener('beforeunload', handleBeforeUnload);
    return () => {
      window.removeEventListener('beforeunload', handleBeforeUnload);
    };
  }, [isDirty]);
}
```

This hook:
- Takes a boolean `isDirty` parameter
- Adds `beforeunload` event listener ONLY when form is dirty
- Cleans up listener on unmount or when isDirty changes to false
- Uses standard browser warning (custom messages blocked by modern browsers)

- Exposes `isDirty` state to parent via a callback prop `onDirtyChange?: (dirty: boolean) => void` (use useEffect to call this when isDirty changes)
- Accepts an `onSuccess` callback to signal successful update (for mode switching back to view)
- Pre-fills initialData from route props:

IMPORTANT: The RouteForm component uses `defaultValue` on its inputs and `useActionState` for form submission. Since RouteForm already handles form state internally via defaultValue, the dirty tracking in RouteEditSection needs to work differently — we can NOT track individual field changes through RouteForm's props.

Instead, implement dirty tracking by:
1. Store a ref to the form element using a wrapper div with a ref
2. Use a MutationObserver or simpler approach: track isDirty as `true` once ANY input/change event fires within the form container (attach an event listener to a wrapper div using event delegation)
3. Reset isDirty to false on successful form submission

The `onDirtyChange` callback lets the parent (RoutePageClient) know when unsaved changes exist, so it can show confirmation on mode toggle.

For onSuccess: RouteForm currently uses useActionState. When state.success becomes true, it currently doesn't do anything visible. The edit-route-client.tsx doesn't handle success either — the updateRoute server action does a redirect. We need to modify this behavior for the unified page:

The existing `updateRoute` action in routes.ts redirects to `/routes/${id}` on success. For the unified page, we want it to stay on the same page but switch to view mode. Two approaches:
a. Change updateRoute to NOT redirect but just revalidate and return success — then the client handles the mode switch
b. Keep the redirect (it will reload the page at /routes/[id] which is the unified page in view mode)

Approach (b) is simpler and works because the redirect goes to `/routes/${id}` WITHOUT `?mode=edit`, which means the page will load in view mode. The redirect will cause a full server re-render, refreshing all data. This is actually the DESIRED behavior after a save — we want fresh data from the server.

So we do NOT need to change the updateRoute action's redirect behavior — it already does the right thing. The onCancel prop is used by the parent to exit edit mode without saving.

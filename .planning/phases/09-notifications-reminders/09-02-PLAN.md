  <action>
    Create `src/app/(owner)/actions/notifications.ts` with 'use server' directive.

    **1. `getUpcomingMaintenance()` server action:**
    - Requires OWNER or MANAGER role (requireRole pattern)
    - Gets tenant-scoped Prisma client via getTenantPrisma()
    - Fetches all active (isCompleted=false) ScheduledService records, including the associated Truck (for name and odometer)
    - Uses `calculateNextDue` from `@/lib/utils/maintenance-utils` to compute due status for each schedule
    - Filters to items where daysUntilDue <= 30 OR milesUntilDue <= 1000 (broader window than email - dashboard shows more context)
    - Also includes items that are already overdue (isDue=true)
    - Sorts: overdue first, then by nearest due date/mileage
    - Returns array of: `{ truckId, truckName (year make model), serviceType, daysUntilDue, milesUntilDue, isDue, nextDueDate, nextDueMileage, currentMileage }`
    - Use @ts-ignore for Prisma 7 withTenantRLS extension type issues (established pattern)

    **2. `getExpiringDocuments()` server action:**
    - Requires OWNER or MANAGER role
    - Gets tenant-scoped Prisma client
    - Fetches all Trucks with select: id, make, model, year, documentMetadata
    - For each truck, parse `documentMetadata` JSONB (type it as `{ registrationNumber?: string, registrationExpiry?: string, insuranceNumber?: string, insuranceExpiry?: string } | null`)
    - Check `registrationExpiry` and `insuranceExpiry` date strings
    - Parse dates, calculate daysUntilExpiry = Math.ceil((expiryDate - now) / msPerDay)
    - Filter for items where daysUntilExpiry <= 60 (broader window for dashboard) OR already expired (daysUntilExpiry < 0)
    - Returns array of: `{ truckId, truckName, documentType ("Registration" | "Insurance"), expiryDate (string), daysUntilExpiry (number), isExpired (boolean) }`
    - Sort: expired first, then by nearest expiry

    **Export types** for both return arrays so the widget components can use them.
  </action>

  </verify>

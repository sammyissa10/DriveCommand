    **1. ExpiryStatusBadge component (`src/components/documents/expiry-status-badge.tsx`):**

    Simple presentational component. Props: `{ expiryDate: Date | string | null }`.

    If expiryDate is null, return null (no badge). Otherwise:
    - Calculate `daysUntilExpiry` using `differenceInDays` from `date-fns` (already installed).
    - Three states:
      - `expired` (daysUntilExpiry < 0): Red badge (`bg-red-100 text-red-800`), text: "Expired {N} days ago"
      - `expiring-soon` (daysUntilExpiry <= 30): Yellow/amber badge (`bg-yellow-100 text-yellow-800`), text: "Expires in {N} days"
      - `valid` (daysUntilExpiry > 30): Green badge (`bg-green-100 text-green-800`), text: "Valid ({N} days)"
    - Use `rounded-full px-2.5 py-0.5 text-xs font-medium` classes (matches existing badge patterns in the codebase).

    **2. DriverDocumentUpload component (`src/components/documents/driver-document-upload.tsx`):**

    `'use client'` component. Props: `{ driverId: string, onUploadComplete: () => void }`.

    This is the most complex component. It handles two upload paths:
    - **Small files (<5MB)**: Use existing single-PUT presigned URL pattern via `requestDriverUploadUrl` + `completeDriverDocumentUpload` server actions.
    - **Large files (>=5MB)**: Use multipart upload via API routes (`/api/documents/multipart/initiate`, `/api/documents/multipart/part-url`, `/api/documents/multipart/complete`).

    **Form fields (shown above the file input):**
    - Document Type: Select dropdown with options "Driver License", "Driver Application", "General Document". Required. Map display names to enum values (DRIVER_LICENSE, DRIVER_APPLICATION, GENERAL).
    - Expiry Date: Date input (`<input type="date" />`). Optional. Shown prominently for DRIVER_LICENSE (pre-label hint: "License documents typically have an expiry date").
    - Notes: Text input. Optional. Max 500 chars. Placeholder: "e.g., State of CA, Class A".

    **File input:** Below form fields. Same drag-and-drop style as existing `DocumentUpload` component. Accept `.pdf,.jpg,.jpeg,.png`. Show "Max 100MB" in the helper text.

    **Upload flow for small files (<5MB):**
    1. User selects file + fills form fields.
    2. Client validates: file size <= 100MB, file type in allowed list, documentType selected.
    3. Call `requestDriverUploadUrl(formData)` — formData includes file, entityType='driver', entityId=driverId, documentType, expiryDate, notes.
    4. PUT file to presigned URL.
    5. Call `completeDriverDocumentUpload(...)` with all metadata.
    6. Reset form and call `onUploadComplete()`.

    **Upload flow for large files (>=5MB) — multipart:**
    1. User selects file + fills form fields.
    2. Client validates same as above.
    3. Calculate `totalParts = Math.ceil(file.size / PART_SIZE)` where `PART_SIZE = 10 * 1024 * 1024` (10MB).
    4. POST to `/api/documents/multipart/initiate` with `{ fileName, contentType, fileSize, entityType: 'driver', entityId: driverId, totalParts }`.
    5. For each part (1 to totalParts):
       a. POST to `/api/documents/multipart/part-url` with `{ s3Key, uploadId, partNumber }`.
       b. Upload chunk using **XMLHttpRequest** (not fetch) to get progress events.
       c. Track `xhr.upload.onprogress` to update progress state.
       d. Extract ETag from response header.
       e. Collect `{ PartNumber, ETag }` for complete step.
    6. POST to `/api/documents/multipart/complete` with `{ s3Key, uploadId, parts, fileName, contentType, sizeBytes, entityType: 'driver', entityId: driverId, documentType, expiryDate, notes }`.
    7. Reset form and call `onUploadComplete()`.

    **Progress UI:**
    - State type: `{ percentage: number, currentPart: number, totalParts: number, uploadedBytes: number, totalBytes: number }`.
    - Show a progress bar (Tailwind: `h-2 rounded-full bg-blue-600` inside `h-2 w-full rounded-full bg-gray-200`).
    - Show percentage text and "Part X of Y" for multipart.
    - Show "Uploading... XX%" during upload, "Saving..." during metadata save.

    **Cancel support:**
    - Store `AbortController` in state.
    - "Cancel" button visible during upload. Calls `controller.abort()`.
    - On abort: show "Upload cancelled" message, clean up.

    **Retry logic:**
    - On part upload failure (non-abort), retry up to 3 times with exponential backoff (1s, 2s, 4s).
    - If all retries fail, show error and allow user to retry the entire upload.

    **Error handling:**
    - Red error banner below the upload area (matching existing `DocumentUpload` error pattern).
    - Clear error on new upload attempt.

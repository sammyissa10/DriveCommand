**src/lib/db/prisma.ts** — Singleton Prisma client:
Create a global Prisma client singleton that prevents multiple instances in development (Next.js hot reload creates new instances). Use the standard `globalThis` pattern:

```typescript
import { PrismaClient } from '@prisma/client';

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient };

export const prisma = globalForPrisma.prisma || new PrismaClient();

if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma;
```

**src/lib/db/extensions/tenant-rls.ts** — RLS Client Extension:
Create a Prisma Client Extension that wraps every query in a transaction setting the `app.current_tenant_id` session variable. This is the critical security layer.

Key implementation details:
- Use `Prisma.defineExtension()`
- Intercept `$allModels.$allOperations`
- Wrap in `prisma.$transaction()` with TWO operations: (1) `$executeRaw` to call `set_config('app.current_tenant_id', tenantId, TRUE)` — the `TRUE` parameter makes it transaction-local, preventing connection pool contamination, (2) the actual query via `query(args)`
- Export function `withTenantRLS(tenantId: string)` that returns the extension
- CRITICAL: The third parameter to `set_config` MUST be `TRUE` (is_local) — this scopes the variable to the current transaction only, not the connection session. This prevents stale tenant context on pooled connections.

Follow the exact pattern from research (Pattern 1 in 01-RESEARCH.md). Do NOT create a new PrismaClient per request — extend the singleton.

**src/lib/context/tenant-context.ts** — Tenant context helper:
Create a utility that extracts tenant ID from the request context. This reads the `x-tenant-id` header that proxy.ts injects.

```typescript
import { headers } from 'next/headers';

export async function getTenantId(): Promise<string | null> {
  const headersList = await headers();
  return headersList.get('x-tenant-id');
}

export async function requireTenantId(): Promise<string> {
  const tenantId = await getTenantId();
  if (!tenantId) {
    throw new Error('Tenant context is required but not found. Ensure proxy.ts is injecting x-tenant-id header.');
  }
  return tenantId;
}
```

Also create a helper that returns a tenant-scoped Prisma client:

```typescript
import { prisma } from '../db/prisma';
import { withTenantRLS } from '../db/extensions/tenant-rls';

export async function getTenantPrisma() {
  const tenantId = await requireTenantId();
  return prisma.$extends(withTenantRLS(tenantId));
}
```

**src/app/proxy.ts** — Next.js 16 proxy (replaces middleware.ts from v15):
Implement the proxy function that intercepts all requests, resolves tenant from Clerk session, and injects the tenant ID as a request header.

Key implementation:
- Export `async function proxy(request: NextRequest)`
- Use `auth()` from `@clerk/nextjs/server` to get the current user's session
- If no userId (unauthenticated), call `NextResponse.next()` — let Clerk's own middleware handle auth redirects
- If userId exists, retrieve `privateMetadata.tenantId` from the Clerk session
- If no tenantId in metadata, check if the request is already going to `/onboarding` or `/api/webhooks` — if not, redirect to `/onboarding`
- If tenantId exists, clone the request headers and set `x-tenant-id` to the tenant ID
- Return `NextResponse.next({ request: { headers: requestHeaders } })`
- Export `config` with `matcher` that excludes: `api/webhooks`, `_next/static`, `_next/image`, `favicon.ico`, `sign-in`, `sign-up`

IMPORTANT per locked decision: Tenant is resolved from auth, NOT from URL/subdomain. No path-based or subdomain-based tenant routing.

Note on Next.js 16: The file is `proxy.ts` NOT `middleware.ts`. The exported function is `proxy` NOT `middleware`. Verify the correct Next.js 16 convention — if `proxy.ts` is not yet supported and still uses `middleware.ts`, use `middleware.ts` with the `middleware` export name instead.

**src/app/api/webhooks/clerk/route.ts** — Clerk webhook endpoint:
Implement the POST handler for Clerk webhook events. This handles tenant provisioning when a user completes email verification.

Key implementation:
- Verify webhook signature using `svix` library (Webhook class)
- Read `svix-id`, `svix-timestamp`, `svix-signature` from request headers
- Parse request body and verify with `wh.verify(body, headers)`
- Handle `user.created` event:
  - Extract `clerkUserId`, `email_addresses`, `public_metadata` from event data
  - Use Prisma to create a new Tenant with: name from `public_metadata.companyName` (or email as fallback), timezone from `public_metadata.timezone` (default to 'UTC')
  - Create a User record linked to the Tenant with role `OWNER`
  - Update Clerk user's `privateMetadata` with `{ tenantId: tenant.id }` using `@clerk/nextjs/server` clerkClient
- CRITICAL: Make the handler IDEMPOTENT — use `upsert` or check if tenant already exists for this clerkUserId before creating. Clerk retries failed webhooks, and there's a race condition between webhook and first login (see Pitfall 6 in research).
- Return 200 on success, 400 on invalid signature, 500 on errors

Use the direct Prisma client (not the RLS-scoped one) since this is a provisioning operation that runs before any tenant context exists. Import from `src/lib/db/prisma.ts`.
